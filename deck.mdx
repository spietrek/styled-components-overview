# CSS vs. CSS-in-JS

| CSS Challenges | CSS-in-JS Benefits |
| ----------------| ------------- |
| CSS development too slow | Full CSS support. Themes. Editor syntax highlighting. Automatic vendor prefixing. |
| Global namespace / Styling conflicts / Specificity | Scoped styles without selectors. Unique class names are created to avoid collisions. Two style definitions for one element can be used to avoid specificity conflicts. |
| Dead code | Linters can show unused code. |
| Hardly reusable / Site-scoped | CSS is defined in *components* which can be reused. |

---

# Component Based Design

1. Components promote reusability.
2. Components can be split UI into small, more manageable parts.
3. Isolating components and states allow developers to articulate behavior and functionality.
4. Components can be developed independently using contract specfications.
5. Component code (including styles) is packaged together to make sharing easier and avoid unused code bloat.
6. Components can be refactored or replaced.
7. Style guides, like Storybook or Styleguidist, allow to work on a component in isolation.
8. [Bit](https://bitsrc.io/). Allows sharing components code between projects. All components are organized on the cloud, made discoverable, and synced to all your projects.

---

> The basic idea of styled-components is to enforce best practices by removing the mapping between styles and components.

<br />

## - Max Stoiber

<Notes>
styled-components Benefits

1. Use as as much or as little as you want.
2. Props instead of classes.
3. Defining styled components that possess their own encapsulated styles.
4. Layout and styling in a single component for efficiency.
5. Highly expressive which makes reading JSX easier (JS + HTML + CSS).

</Notes>
